# 删除策略修订实施总结

## 概述

本次修订针对项目中删除操作存在的数据一致性问题，实施了**方案 A + 增强监控**策略，确保删除操作的可靠性和数据一致性。

---

## 问题回顾

### 原始删除机制的问题

1. **乐观更新无等待确认**
   - 前端立即更新 UI，但后端删除是异步的
   - 如果 db-server 删除失败，用户已看到删除成功的提示

2. **MinIO 清理静默失败**
   - MinIO 删除失败只在 console.warn，用户完全不知道
   - 导致"数据库删了，但 MinIO 文件还在"

3. **localStorage 优先级过高**
   - 页面刷新时优先从 localStorage 加载
   - 如果 localStorage 有旧数据，用户看到的是过时数据

4. **缺少事务性保障**
   - 删除涉及 3 个独立系统（前端 state、db-server、MinIO）
   - 没有事务机制，任何环节失败都会导致数据不一致

---

## 实施方案

### 1. 同步删除（SourceMaterialsPage.tsx）

**修订内容：**
- `handleDelete`: 单个删除改为同步等待
- `handleBatchDelete`: 批量删除改为同步等待
- `handleClearAll`: 清空全部改为同步等待

**删除流程：**
```
用户点击删除
    ↓
等待确认对话框
    ↓
调用 db-server DELETE /materials
    ↓ [失败] → 弹错误提示，不更新 UI
    ↓ [成功]
调用 upload-server DELETE /delete-material（清理 MinIO）
    ↓ [失败] → 弹警告"数据库已删除，但云存储清理失败"
    ↓ [成功]
更新前端 state + localStorage
    ↓
显示成功提示
```

**关键代码变更：**
```typescript
// 同步删除：先调用后端 API
try {
  const { dbDelete } = await import('../../store/appContext');
  await dbDelete('/materials', { ids: [id] });

  // 清理 MinIO
  if (material?.metadata?.provider === 'minio') {
    const res = await fetch('/__proxy/upload/delete-material', {
      method: 'POST',
      // ...
    });
    if (!res.ok) {
      toast.error('数据库已删除，但云存储清理失败，可能存在残留文件');
    }
  }

  // 全部成功后更新前端 state
  dispatch({ type: 'DELETE_MATERIAL', payload: [id] });
  toast.success('已删除');
} catch (err) {
  toast.error('删除失败，请重试');
}
```

---

### 2. MinIO 清理明确提示（appContext.tsx）

**修订内容：**
- 移除 appContext 中的自动 MinIO 清理逻辑
- 改为日志记录：`MinIO cleanup handled by delete handler`

**原因：**
- 现在删除操作由 SourceMaterialsPage 统一控制（同步删除）
- 避免 MinIO 清理逻辑分散在两处

---

### 3. localStorage 优先级调整（appContext.tsx + types.ts）

**修订内容：**
1. 在 `AppState` 接口添加 `_dataSource` 字段
2. 初始化时标记数据来源为 `'localStorage'`
3. db-server hydration 成功后更新为 `'db-server'`

**类型定义：**
```typescript
export interface AppState {
  // ... 其他字段
  _dataSource?: 'localStorage' | 'db-server' | 'initial';
}
```

**初始化代码：**
```typescript
const initialState: AppState = {
  // ...
  _dataSource: 'localStorage',
};

// db-server hydration 后
dispatch({
  type: 'HYDRATE_FROM_DB',
  payload: {
    // ...
    _dataSource: 'db-server',
  },
});
```

---

### 4. 数据一致性检查工具（SettingsPage.tsx）

**新增功能：**

#### 4.1 新增"数据一致性检查" Tab

在设置页面添加新的 tab，包含以下工具：

**A. 数据来源检查**
- 显示当前数据来源（localStorage / db-server / initial）
- 如果是 localStorage，显示警告提示
- 提供"强制刷新数据"按钮

**B. 数据库与存储一致性**
- 显示数据库资料数
- 显示 MinIO 对象数
- 说明常见原因和检测方法

**C. MinIO 孤儿文件检测**
- 调用 `GET /__proxy/upload/check-orphaned-files`
- 检测 MinIO 中存在但数据库无记录的文件
- 显示检测结果和文件列表

**D. 清理工具**
- 提供清除 localStorage 缓存按钮
- 清除后自动刷新页面，从 db-server 重新加载数据

---

### 5. upload-server 新增诊断 API

#### 5.1 GET /check-orphaned-files

**功能：**
- 从 db-server 获取所有 materials
- 收集所有有效的 MinIO 对象路径
- 列出所有 MinIO 对象
- 检测孤儿文件（存在但数据库无对应记录）

**返回数据：**
```json
{
  "ok": true,
  "orphaned": [
    {
      "name": "originals/12345/file.pdf",
      "size": 1024000,
      "lastModified": "2026-04-15T00:00:00Z"
    }
  ],
  "totalObjects": 100,
  "validObjects": 95
}
```

#### 5.2 POST /repair-consistency

**功能：**
- 接收孤儿文件列表
- 批量删除孤儿文件
- 返回删除结果和错误信息

**请求体：**
```json
{
  "orphanedFiles": [
    { "name": "originals/12345/file.pdf" }
  ]
}
```

**返回数据：**
```json
{
  "ok": true,
  "removed": 5,
  "errors": []
}
```

## 总结

本次修订通过以下改进，显著提升了删除操作的可靠性和数据一致性：

1. ✅ **同步等待**：删除操作改为同步等待后端确认

5. ✅ **localStorage 降级**：优先使用 db-server 数据
