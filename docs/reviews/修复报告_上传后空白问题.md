# 修复报告：上传后空白问题 (Material 未持久化)

## 问题描述

**现象**: 上传 FastTest01.pdf → MinIO 成功 → 界面空白，material 未出现在列表中

**根本原因**:
- 前端 `BatchUploadModal.tsx` 上传流程只做了两件事:
  1. `dispatch({ type: 'ADD_MATERIAL' })` → 仅写入 React 内存状态
  2. `POST /upload` → 上传文件到 MinIO ✅
  3. `POST /batch/jobs` → 调用空壳 stub（返回 `{ok:true, added:0}`）
- **缺失的关键步骤**: 没有调用 db-server 创建 material 记录
- 旧的 `addJobs()` 原本会同步创建 db 记录，但简化后的 `batch-queue.mjs` 中它变成了纯空壳
- 结果: material 只存在于 React 内存，刷新页面就消失

**影响范围**:
- 所有新上传文件都不会出现在资料列表
- 前端 "解析" / "AI分析" 按钮无法工作（没有 material ID 可用）

## 解决方案

**选择方案**: 方案 B (前端直驱)

**理由**:
1. 符合之前的架构决策 - 已移除 Batch Queue 架构，改用 Material 直驱模式
2. 代码更清晰 - 前端直接调用 db-server `/materials` 端点，不需要经过废弃的队列
3. 减少依赖 - 不需要在旧的 `/batch/jobs` 端点中"复活"数据库创建逻辑
4. 一致性 - 与"解析"按钮的调用方式保持一致

## 技术细节

### db-server `/materials` 端点
- 位置: `/workspace/ops/Luceon2026/server/db-server.mjs` 第 261-267 行
- 要求请求体包含 `id` 字段
- 返回: `{ ok: true, id: <传入的ID> }`
- 数据写入: `dbCache.materials[id] = item` 并调用 `writeDB()` 持久化

### ID 管理
- 前端使用时间戳 + 计数器生成唯一临时 ID
- db-server 直接使用前端提供的 ID，不做额外分配
- 这样可以确保前后端 ID 一致，无需同步更新

### 错误处理
- MinIO 上传失败: 抛出异常，更新 material 状态为 `failed`
- 数据库创建失败: 抛出异常，更新 material 状态为 `failed`
- 任何失败都会显示 toast 通知

- Material 记录从内存存储迁移到数据库存储
