# 一致性清理与历史数据运维操作说明

本指南旨在指导开发与运维人员安全地使用 `/audit/consistency` 接口进行历史脏数据审计与清理。

## 一、核心原则

1. **预览优先**：在执行任何删除或更新操作前，必须先通过 `GET` 请求查看审计报告。
2. **备份先行**：在执行 `apply` 前，建议通过 `GET /__proxy/db/backup/full-export` 导出完整资产备份。
3. **不可逆性**：`orphan-object`（孤儿文件）的删除操作是直接调用 MinIO API 的，删除后无法找回，请务必确认文件确实无用。

## 二、审计入口

### 1. 启动 Dry-run 扫描
**Endpoint:** `GET /__proxy/upload/audit/consistency`

该接口会返回一个 JSON 报告，包含以下关键信息：
- `findings`: 发现的问题清单，每项包含 `kind`, `message`, `suggestion` 以及预设的 `repair` 动作。
- `counters`: 统计信息（素材总数、任务总数等）。

## 三、问题类型（Finding Kinds）解析

| 问题类型 | 严重程度 | 说明 | 建议处理方式 |
| :--- | :--- | :--- | :--- |
| `orphan-task` | Error | 任务引用的素材 ID 在数据库中不存在。 | 安全。可标记为 `failed` 或删除。 |
| `orphan-ai-job` | Error | AI Job 引用的解析任务 ID 不存在。 | 安全。建议清理。 |
| `original-file-missing` | Error | 数据库记录了文件，但 MinIO 物理文件已丢失。 | **警告**。该资料已损坏。`apply` 会执行级联删除。 |
| `original-file-missing-with-completed-task` | Error | 原始文件丢失，但关联任务已完成（有 AI 结果）。 | **建议**。标记任务为 `resourceLost`，保留 AI 结果供参考。 |
| `parsed-file-missing` | Warn | 解析产物（Markdown）丢失。 | 推荐。标记为资源缺失或触发重解析（需原始文件存在）。 |
| `orphan-object` | Warn | MinIO 中存在文件，但数据库无任何记录。 | **审慎**。涉及物理删除，建议人工确认后选择性清理。 |
| `dangling-material-ai-job` | Warn | 素材引用的 AI Job ID 不存在。 | 安全。重置素材 AI 状态为 `pending`。 |
| `ai-running-without-active-job` | Warn | 任务处于 `ai-running` 但无活跃 AI Job。 | 安全。重置为 `ai-pending` 触发重新识别。 |
| `non-canonical-task-state` | Info | 任务使用了过时的状态名（如 `succeeded`）。 | 安全。自动迁移至新版 Canonical 状态。 |
| `non-canonical-ai-state` | Info | AI Job 使用了旧版/非规范状态名。 | 安全。自动迁移至 Canonical AI 状态。 |
| `bad-original-prefix` | Info | 原始文件路径不符合 `originals/{id}/` 规范。 | **人工**。仅告警，需手动确认是否迁移。 |
| `bad-parsed-prefix` | Info | 解析产物路径不符合 `parsed/{id}/` 规范。 | **人工**。仅告警，需手动确认。 |

## 四、示例 Dry-run 分布说明

以下为 Lucia 在某次验收环境中的扫描示例（仅供参考，不代表固定结果）：

- **orphan-object**: 37 (大量上传中断留下的残留)
- **orphan-task**: 4 (手动删除素材后留下的残留任务)
- **parsed-file-missing**: 3 (解析失败或手动删除导致的产物缺失)
- **bad-parsed-prefix**: 2 (历史版本留下的不规范路径)

## 五、执行清理（Apply）

### 1. 全量修复
**Endpoint:** `POST /__proxy/upload/audit/consistency/apply`
**Body:** `{}`

执行扫描出的所有带 `repair` 动作的修复项。**注意：不建议在包含大量 `orphan-object` 时直接全量执行。**

### 2. 选择性修复
**Endpoint:** `POST /__proxy/upload/audit/consistency/apply`
**Body:**
```json
{
  "findings": [
    { "kind": "orphan-task", "targetId": "task-123", "repair": { ... } }
  ]
}
```
仅执行传入的特定项。建议从前端 UI 勾选后提交。

## 六、安全边界

- **对象存储保护**：目前 `orphan-object` 扫描仅覆盖 `originals/` 和 `parsed/` 前缀。请勿在此目录外手动放置重要文件。
- **操作建议**：`orphan-object` 的删除操作是不可逆的。执行前必须先进行数据库及文件备份，并避开系统上传/解析的高峰时段。
- **并发控制**：执行 `apply` 时请确保没有大规模的上传任务正在进行，以免误删刚上传但尚未落库的临时文件。
- **超时处理**：扫描大型仓库时可能需要 30s+，请调整客户端超时设置。
